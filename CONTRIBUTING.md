# CONTRIBUTING — 怎么帮这个 skill 变好

这个项目最需要的贡献,不是写新代码,而是**当 Boss直聘改版、选择器腐烂时,帮忙定位失效点、提供新选择器**。因为它驱动的是活的网站,选择器会随平台改版失效——社区多一双眼睛,就多一分"改版当天就修好"的机会。

## 贡献的主要形式(按价值排)

1. **报告 + 修复失效的选择器**(最高价值)——Boss 改版后某个操作点不动了,定位是哪个选择器/流程坏了,给出新的。
2. **补齐"待补"项**——见 `operation-map.md` §8 的清单(如 约面流程、搜索/消息 XHR 端点)。
3. **实现 ROADMAP 里的功能**——见 `ROADMAP.md`,认领一条开 issue。
4. **文档 / FAQ / 故障排查**——你踩过的坑,写下来就是别人的路标。

## ⚠ 前置:验证需要真实账号

改选择器**必须在真机上验证**——你需要:
- 一个已登录的 Boss直聘**招聘者**账号(企业版,能看推荐牛人/搜索牛人);
- browser-act CLI + chrome-direct(见 README「用前提」)。
> 没法验证的 PR 我们也欢迎,但请在描述里注明"未实测",维护者会在有账号时复验。

## 报告一个失效的选择器(issue 模板)

开 issue 时请包含:
```
### 失效的操作
（例:搜索页点「打招呼」没反应 / 推荐列表读不到候选人)

### 复现
1. 走到哪个页面(URL)
2. 执行了什么(browser-act 命令 / playbook 哪一步)
3. 预期 vs 实际

### 定位到的选择器 / 报错
（operation-map 里哪一条?browser-act state / eval 的输出?)

### 建议的新选择器(如已找到)
（新的 CSS 选择器 / 索引取法,最好附 browser-act state 片段佐证)

### 环境
- Boss 改版日期(大概)/ 你的验证时间
- browser-act 版本
```

## 诊断一个"点不动了"的一般步骤

1. `browser-act --session <名> state` 看目标元素还在不在、`[index]` 是多少;
2. 若 state 抓不到,用 `eval` 读 `contentDocument`(注意 iframe:推荐/搜索前默认视图在 `recommendFrame`,搜索结果在 `searchFrame`)确认元素是否改了类名/结构;
3. 区分是**选择器变了**、还是**流程门槛变了**(如 求简历要候选人先回复才 enabled);
4. 找到新选择器后,在 `operation-map.md` 对应条目更新,并注明"YYYY-MM-DD 实测";
5. 若能抓到更稳的 **XHR 接口**,优先记接口(见 §7e),比选择器抗腐烂。

## PR 约定

- **一个 PR 一件事**(一个失效点 / 一个功能),标题写清"修 X 选择器"或"加 Y 功能"。
- 改了 `operation-map.md` 的选择器,请注明**实测日期**和**佐证**(state/eval 片段)。
- 更新了执行层,记得同步 `playbook.md` 里对应的引用(两者交叉引用要对得上)。
- **🔴 别提交任何真实候选人/账号数据**:`ledger.jsonl`、非示例 `reports/*.md`、简历附件、截图——`.gitignore` 已默认忽略,但提交前请再 `git status` 扫一眼。示例数据一律放 `*-example/` 且用 `*.example` 文件名。

## 行为准则

对候选人数据负责、对招聘对象尊重;别把本项目用于骚扰、歧视性筛选或平台条款禁止的用途。安全红线见 `SAFETY.md`。
